# tax-claw — дизайн

**Дата:** 2026-06-15
**Статус:** черновик дизайна (ожидает финальной вычитки пользователем)
**Тип:** TUI-приложение — LLM-агент для подготовки чешской декларации по подоходному налогу физлиц

---

## 1. Краткое описание

`tax-claw` — это терминальный (TUI) ассистент на базе LLM, который помогает подготовить
чешскую декларацию по подоходному налогу физических лиц (*Přiznání k dani z příjmů
fyzických osob*, формуляр 25 5405). Вдохновлён UX агентных приложений вроде Claude Code /
OpenCode: чат-цикл, использование инструментов, память, генерация инструментов «на лету».

Целевой профиль доходов (v1): **зарплата (employment) + вестинг RSU + дивиденды США +
продажа акций**, с учётом договора об избежании двойного налогообложения US–CZ и зачёта
иностранного налога (*zápočet*).

### Главный принцип

> **Агент оркеструет, детерминированный код считает.**
> LLM читает документы, классифицирует, ищет нормы, объясняет и ведёт диалог.
> Всю арифметику делают сгенерированные, протестированные и одобренные функции,
> исполняемые детерминированно. **Агент никогда не складывает числа с плавающей точкой
> «в уме».** Каждая итоговая цифра прослеживается до источника (документ + цитата § закона
> или строка официальной инструкции) через структурную трассировку расчёта.

### Целевая аудитория

Личный инструмент для меня и нескольких человек в похожей ситуации (например, коллеги с RSU).
**НЕ** массовый SaaS-продукт. Это влияет на решения: модель данных чуть гибче, чем хардкод
под один кейс, но без мультиюзерных продуктовых сложностей.

---

## 2. Объём v1 и явные не-цели

### В объёме v1
- Создание проектов-деклараций по налоговому году (например, «Декларация 2027»).
- Кросс-проектный профиль, переподтверждаемый в каждом новом проекте.
- Приём документов в чат, автоопределение типа, извлечение данных (включая сканы/фото через OCR/Vision).
- Поиск по чешскому законодательству (RAG) для расчётов и свободных вопросов.
- Парсинг официальной формы 25 5405 и инструкции по заполнению (*pokyny k vyplnění*).
- Генерация, тестирование, одобрение и кэширование функций расчёта.
- Память: профиль, факты проекта, фидбэк/исправления, выученные артефакты, предпочтения.
- Экспорт **по нарастающей**: (1) Summary с трассировкой → (2) PDF формы → (3) XML для портала.
- Раздельное хранилище (приватное vs шериемое) и формат skill/knowledge pack для шеринга.

### Не-цели v1 (осознанный YAGNI)
- Отдельный механизм «plugins» сверх skills + MCP.
- Маркетплейс/реестр пакетов (используем git-репозиторий + PR).
- Автоматическая подача декларации в портал от имени пользователя.
- Типы доходов вне профиля выше (аренда, ИП/OSVČ, крипта и т.п.) — добавляются позже как skills.
- Мультиюзерные продуктовые возможности (роли, облачная синхронизация).

## 3. Технологический фундамент

| Слой | Решение | Назначение |
|---|---|---|
| Провайдер LLM | **Microsoft.Extensions.AI** (`IChatClient`) | Провайдер-агностичная абстракция: Azure OpenAI / OpenAI / Anthropic / Ollama |
| Агентное ядро | **Microsoft Agent Framework** (`Microsoft.Agents.AI`) | tool-calling, память (context providers), middleware, сессии/checkpoint, human-in-the-loop, Agent Skills, MCP-клиент |
| Инструменты | **MCP C# SDK** (`ModelContextProtocol`) | единый протокол для внутренних и внешних инструментов |
| TUI | **Spectre.Console** | чат-скролл, таблицы строк декларации, промпты, прогресс |
| Платформа | **.NET** (последняя LTS/current) | язык реализации |

**Почему MAF:** даёт из коробки память, провайдер-агностичность, middleware (идеально для
PII-границы), MCP и человеко-в-цикле — то, что иначе пришлось бы переизобретать. Доменный
слой (расчёт, форма, законы, аудит) пишем сами для полного контроля.

> Риск: MAF относительно новый, API ещё развивается. Митигируем изоляцией доменного слоя за
> собственными интерфейсами, чтобы смена версии/фреймворка не затрагивала бизнес-логику.

## 4. Архитектура

```
Пользователь ⟷ TUI (Spectre.Console)
                 ⟷ Агент (Microsoft Agent Framework)
                      │  middleware: PII-граница (редактирование/восстановление)
                      │      ⟷ IChatClient ⟷ LLM-провайдер (Azure OpenAI / OpenAI / Anthropic / Ollama)
                      │
                      ├─ Инструменты (MCP / native):
                      │    classify/extract документов, генерация парсера+песочница,
                      │    law-RAG (поиск нормы), decimal-примитивы, генерация функций расчёта,
                      │    get_fx_rate (ČNB), remember/profile, render_output
                      │
                      └─ Доменные сервисы (детерминированный C#):
                           среда исполнения функций расчёта + аудит-трейл,
                           модель формы 25 5405 (DAG строк),
                           корпус законодательства (RAG, версии по редакциям),
                           FX-сервис, пайплайн документов,
                           экспортеры: Summary → PDF → XML

Локальное хранилище (шифрование at-rest):
   приватное: профиль, проекты, документы, декларации
   шериемое:  skills/knowledge packs (парсеры, функции, правила)
   память:    фидбэк, выученные артефакты, предпочтения
```

### Слои и ответственность
- **TUI** — чат, приём документов, таблицы строк декларации, промпты, прогресс.
- **Агент (MAF)** — системный промпт с ролью и guardrails; tool-calling; память через
  context-providers; PII-middleware; провайдер через `IChatClient`.
- **Инструменты** — тонкая прослойка: агент вызывает доменные сервисы, а не считает сам.
- **Доменные сервисы** — детерминированный, тестируемый код. Источник истины для цифр.
- **Хранилище** — прозрачные файлы + SQLite для индексов/памяти; шифрование через OS keychain.

## 5. Модель данных и хранилище

### Сущности
- **Profile (кросс-проект):** ФИО, rodné číslo / DIČ, адрес, finanční úřad, банковский счёт для
  возврата, статус налогового резидента, работодатель, брокеры. Переподтверждается в каждом
  новом проекте и «вмораживается» снапшотом в проект.
- **Project (год):** привязан к налоговому году; хранит снапшот профиля, документы,
  извлечённые данные, посчитанную декларацию, аудит-лог, версию законодательства и формы.
- **TaxReturn (каноническая модель):** формат-независимый единый источник правды. Источники
  дохода (employment, события вестинга RSU, дивиденды, продажи бумаг), вычеты, налоговые
  кредиты (зачёт иностранного налога), итоги, маппинг на строки формы (*řádky*), `Derivation`
  на каждую цифру, аудит-записи.

### Раскладка проекта
```
~/.tax-claw/
  profile.json            # кросс-проектный профиль (шифр.)
  memory/                 # память + фидбэк + предпочтения
  skills/                 # установленные skills / knowledge packs (шериемые артефакты)
  law/                    # корпус законодательства, версии по редакциям
  projects/2027/
    project.json          # метаданные, статус, версия профиля, версии закона/формы
    session.ckpt          # checkpoint диалога агента (возобновление)
    worksheet.json        # чеклист сбора, открытые вопросы, неподтверждённое
    documents/            # оригиналы (шифр.)
    extracted/            # структурированные JSON на документ
    return.json           # каноническая модель + Derivation на каждую цифру
    calculation-trace.json# полная трассировка расчётов (для «объясни»)
    audit.log             # хронология: цифра → источник + §
    outputs/              # summary / PDF / XML
```

### Жизненный цикл проекта
`черновик → сбор документов → извлечение/проверка → расчёт → ревью → экспорт → подано`

### Промежуточное состояние и возобновление
- **Сессия агента** (`session.ckpt`) — checkpoint диалога MAF, персистится per-project.
- **`worksheet.json`** — рабочее состояние: чеклист, открытые вопросы, неоднозначные/
  неподтверждённые позиции, флаги «нужна проверка». Отделено от итоговой модели.

### Объяснимость
Среда расчёта выдаёт на каждую строку структурный `CalculationTrace`:
> вход (документ + значения) → применённая функция/правило (со ссылкой на § и/или строку
> инструкции) → промежуточные шаги (напр. USD→CZK по конкретному курсу ČNB и дате) → результат.

На вопрос «как получилась сумма в řádku N?» система разворачивает трассировку, а LLM
пересказывает её, **не пересчитывая**.

## 6. Пайплайн документов и генерация инструментов

### Поток
```
Документ в чат (PDF / скан / фото / CSV / XLSX / HEIC)
  → классификация (тип + уверенность); низкая → спросить пользователя
  → определить наличие текстового слоя:
        есть текст → прямое извлечение
        скан/фото/image-PDF → OCR-путь
  → извлечение сущностей (entities) по схеме типа документа
  → если формат известен и есть парсер → детерминированный парсер
  → если формат новый → LLM-извлечение; если формат повторяющийся → генерация парсера
        → показать код + одобрение → песочница → валидация схемы + sanity
  → маппинг в каноническую модель (+ provenance документа)
  → проверка человеком: подтверди / исправь
        исправления → память (фидбэк)
        одобренный парсер → память (выученный артефакт + hash)
```

### Изображения, OCR и извлечение сущностей
Документ может прийти фото/сканом или PDF без текстового слоя. Два режима распознавания
(часть провайдер-абстракции):
- **Vision-LLM** (Azure OpenAI / OpenAI / Anthropic с vision) — изображение → текст + сущности
  сразу; лучшее качество; проходит через PII-слой.
- **Локальный OCR** (Tesseract / OS Vision) — приватный/оффлайн режим: текст локально, затем
  извлечение сущностей.

Под каждый тип документа — целевая схема сущностей. Примеры:
- *RSU vesting statement* → дата вестинга, кол-во акций, FMV/курс акции, gross/net, удержанный налог.
- *Dividend statement* → эмитент, дата выплаты, брутто, налог у источника, валюта.
- *Výpis o zdanitelných příjmech* (от работодателя) → доход, удержанный налог/авансы, взносы.

Сущности извлекаются **строго по схеме** (обуздание prompt-injection), с привязкой к
странице/bounding-box для прослеживаемости. Незаполненные обязательные поля → уточняющий вопрос.

### Безопасность генерации кода
- **Песочница:** изолированный подпроцесс/контейнер — без сети, ФС ограничена (целевой
  документ на чтение + temp на запись), лимиты CPU/RAM/времени.
- **Гейт одобрения:** сгенерированный код показывается пользователю перед первым запуском
  (настройка: всегда спрашивать / авто-запуск только в песочнице).
- **Пиннинг:** одобренный парсер хешируется и переиспользуется; вывод обязан пройти валидацию
  схемы и sanity-проверки.
- **Документы — недоверенный ввод:** извлечённый текст/изображение трактуются как **данные,
  не инструкции**.
- Язык парсеров — Python (pdfplumber/pandas) или C# (PdfPig); песочница поддерживает оба.

## 7. Расчёт: законодательство, форма, сгенерированные функции

### 7.1. Закон как searchable knowledge base (RAG), без хардкода

Агент **не знает заранее**, какие параграфы понадобятся. Любой вопрос — про расчёт или
«поясни норму X» — решается через поиск по проиндексированному корпусу.

```
Корпус (индексируется целиком, не хардкодится):
  586/1992 Sb. о подоходном налоге + связанные акты
  + договор US–CZ об избежании двойного налогообложения
  + методики GFŘ
  → версии по редакциям (дата вступления + хеш)
  → разбивка на §/абзацы + метаданные
  → гибридный индекс: полнотекст + векторы + граф перекрёстных ссылок между §

Единый retrieval (для расчёта И свободных вопросов):
  запрос → keyword + семантика → реранкинг + раскрытие перекрёстных ссылок
        → топ-N релевантных § с текстом и ссылками (НЕ весь закон в контекст)
```

- **Никакого фиксированного списка § и захардкоженных ставок.** Расширяемость: новый тип
  дохода или новый закон = новые документы в корпусе, **без правки кода**.
- **Единый механизм поиска** обслуживает и расчёт, и свободные вопросы про чешское налоговое
  право (со ссылкой на конкретный § и редакцию).
- **Актуальность «на момент подачи»:** для проекта-года выбирается действующая на момент
  подачи редакция; и расчёт, и пояснения опираются на правильную версию.

### 7.2. Официальная форма и инструкция как второй источник правил

Скачиваем и парсим (тем же пайплайном документов) **форму 25 5405 + pokyny k vyplnění**.
Чешские *pokyny* буквально описывают построчные формулы («řádek 38 = řádek 36 − řádek 37») и
ссылки на §. Структура формы становится **графом расчёта (DAG)** по строкам.

### 7.3. Расчёт через сгенерированные функции (без float в «уме» агента)

```
Источники правил: законодательство (RAG) + форма/инструкция (DAG строк)
  → агент выводит ПРАВИЛО для строки/величины
  → есть одобренная функция в кэше (версии совпадают)?
        да → исполнить
        нет / закон или форма изменились →
            агент ГЕНЕРИРУЕТ функцию (decimal, без float)
            → автотесты + sanity на граничных случаях
            → показать код + норму-первоисточник на одобрение
            → кэш функций (пиннинг: § + редакция + версия формы)
  → песочница: детерминированное исполнение
  → CalculationTrace: входы → функция → § → результат
  → каноническая модель + строки формы + аудит
```

- **Агент не делает арифметику сам.** Любой расчёт = вызов сгенерированной функции,
  исполняемой детерминированно. Деньги — `decimal`/целочисленная копеечная арифметика,
  **никогда** бинарный float.
- **Функции генерируются, тестируются, одобряются, кэшируются** и пиннятся к версии закона +
  версии формы. Со временем накапливается доказуемо корректная библиотека правил.
- **Полная прослеживаемость:** каждая функция несёт провенанс (§ + строка формы +
  редакция/хеш); исполнение пишет `CalculationTrace`.

### 7.4. Примитивы decimal-математики

Минимальный набор инструментов: `add`, `subtract`, `multiply`, `divide`, `round`. `round`
реализует официальное чешское *zaokrouhlení* (база — вниз до сотен Kč, налог — до целых Kč и
т.п.). Всё в `decimal`/целочисленных минорных единицах, **никогда** float.

**Два слоя с чёткой границей:**
1. **Примитивы** — единственный разрешённый способ для агента получить любое число «на лету»
   (жёсткий guardrail против устного float-счёта). Для ad-hoc (конвертация одного дивиденда
   USD→CZK, sanity-проверки) и как матядро/правила округления для генерируемых функций.
2. **Сгенерированные функции** — формальный построчный расчёт (DAG формы), переиспользуемые,
   с тестами, провенансом и трассировкой.

**Граница:** формальный расчёт строки декларации **всегда** идёт через именованную функцию.
Длинная цепочка `add/add/multiply` в чате для расчёта строки — красный флаг (обходит слой
тестируемых/одобряемых/кэшируемых функций и теряет провенанс).

### 7.5. Покрытие правил v1 (профиль доходов)
- **§ 6 — employment + RSU:** зарплата по výpis; вестинг RSU как доход от занятости в момент
  вестинга (FMV × курс ČNB на дату).
- **§ 10 — продажа акций:** прирост = выручка − база (база = FMV на вестинге); проверка
  временно́го теста освобождения; FIFO при необходимости.
- **§ 8 — дивиденды:** брутто + налог, удержанный у источника в США.
- **Зачёт иностранного налога (*zápočet*)** по договору US–CZ: ограничение зачёта/maximum.

## 8. Память и агентный цикл

### Тиры памяти (по сроку жизни и доверию)
- **Профиль** — кросс-проект, живёт годами, переподтверждается в каждом проекте.
- **Факты проекта-года** — внутри одной декларации.
- **Фидбэк/исправления** — поправки пользователя («RSU от Microsoft трактуй как § 6», «это
  return of capital, не дивиденд»). **Приоритет над дефолтным поведением** агента.
- **Выученные артефакты** — одобренные парсеры и функции расчёта; переиспользуются без LLM.
- **Предпочтения** — язык, провайдер LLM, режим приватности.

### Как фидбэк делает систему точнее
Правка сохраняется с областью действия (этот проект / все будущие / конкретный тип документа)
и в следующий раз подаётся агенту как контекст через MAF context-provider. Спорные места агент
выносит на подтверждение, а не угадывает молча.

### Инвалидация по версии
При изменении закона/формы выученные функции **инвалидируются по версии** (пиннинг из §7),
чтобы прошлогоднее правило не применилось к новому году молча.

### Агентный цикл (на каждом ходе)
```
ввод/документ
  → подтянуть память (профиль + фидбэк + факты)
  → спланировать шаг
  → вызвать инструмент (не считать в уме)
  → уверенность?
        низкая/спорно → спросить пользователя (human-in-the-loop) → записать фидбэк
        высокая → обновить модель + трасса
  → checkpoint сессии + worksheet
```

## 9. Расширяемость: skills и MCP (без plugins)

Решение: **YES skills, YES MCP, NO отдельный plugin-механизм (YAGNI).** MAF поддерживает оба
нативно (Agent Skills + MCP-клиент).

- **MCP = единый протокол инструментов («руки»).** Внутренние тулы (law-RAG, генерация
  функций, примитивы, экспорт) и будущие внешние серверы (курсы ČNB, доступ к закону) говорят
  на одном протоколе. В v1 внешних серверов почти нет; ценность — единый интерфейс на будущее.
- **Skills = пакеты знаний и правил («навыки»), и это те же knowledge packs (см. §10).**
  Скилл — сценарная единица («вестинг RSU §6», «дивиденды США с withholding», «заполнение
  řádku 38»): парсер формата + функции расчёта + правила трактовки + инструкция +
  провенанс/версия. Агент подгружает релевантный скилл по необходимости (а не держит всё в
  системном промпте).
- **Plugins** как третий механизм — **не добавляем** (дублирование; MCP-тулы + skills +
  генерируемые функции покрывают расширяемость).

### Безопасность расширений
- **Скиллы недоверенные, как код.** Текст-инструкция в чужом скилле может содержать
  prompt-injection. Защита: любая цифра обязана прослеживаться до корпуса закона (цитата §)
  или до протестированной функции — врущий скилл не сможет «назначить» ставку без проверяемого
  первоисточника. Импорт проходит: песочница + тесты + одобрение + пиннинг версии.
- **Внешние MCP-серверы** — недоверенные данные/код: явное подключение, least-privilege, та же
  PII-граница.

## 10. Шеринг улучшений (knowledge packs)

Чтобы делиться улучшениями, выученные артефакты **отделены от приватной памяти by design**.

### Разделение хранилища (с первого дня)
- **Приватное** (никогда не покидает машину): профиль, rodné číslo, суммы, документы,
  фактические декларации.
- **Шериемое** (обобщённое, без PII): парсеры форматов документов, функции расчёта строк
  (+ тесты + провенанс), правила трактовки/маппинга (напр. RSU MSFT → § 6).

PII не может утечь через пакет **по конструкции**; плюс перед экспортом — авто-сканер на
остаточные PII.

### Формат и канал
- **Knowledge pack = skill** (см. §9): версионированный набор артефактов с манифестом (привязка
  к версии закона/формы, автор, хеш, тесты).
- **Канал:** общий **git-репозиторий** скиллов, вклад через **PR + ревью**. Аудируемо,
  сохраняет провенанс, естественно для разработчика. Маркетплейс/реестр — НЕ делаем (YAGNI).

### Безопасность импорта (критично)
Импортируемый парсер/функция — **чужой исполняемый код**. Проходит тот же путь, что и
сгенерированный агентом: песочница (без сети, лимиты) + прогон тестов + **явное одобрение**
пользователем перед первым использованием. Никакого слепого доверия, даже если тесты зелёные.
Версионная привязка сохраняется: при несовпадении года/редакции агент адаптирует с
подтверждением или генерирует заново — чужое правило не применяется к твоему году молча.

## 11. Сквозные слои: приватность, безопасность, экспорт, ошибки

### Приватность (PII-граница как middleware MAF)
- **Локальный режим** (Ollama): PII никуда не уходит.
- **Облачный режим:** middleware на границе детектирует PII (rodné číslo, суммы, имена, счета),
  заменяет на токены-плейсхолдеры (псевдонимизация) и восстанавливает их в ответе. Провайдер
  видит структуру задачи, но не персональные данные.
- **At-rest:** шифрование через OS keychain.

### Безопасность исполнения (сведено воедино)
- Любой неродной код (генерированные парсеры/функции, импортированные скиллы) — в **песочнице**:
  без сети, ограниченная ФС, лимиты CPU/RAM/времени.
- Документы, скиллы, внешние MCP — **недоверенный ввод**: данные, не инструкции; защита
  провенансом (цифра → § или тестированная функция).
- Гейт одобрения + пиннинг к версии закона/формы.

### Экспорт (по нарастающей — три экспортера поверх канонической модели)
1. **Summary** — постро́чный расчёт с трассировкой и ссылками на § (вехой v1).
2. **PDF** формы 25 5405 — заполненный, для проверки/печати/приложения.
3. **XML** для портала MOJE daně / EPO, с **валидацией против официальной XSD-схемы**.

Каждый — отдельная веха; каноническая модель — единый источник правды для всех трёх.

### Ошибки и доверие
- Низкая уверенность → вопрос пользователю (не молчаливая догадка).
- Пробелы и конфликты данных подсвечиваются в `worksheet.json`.
- Везде дисклеймер: **инструмент-помощник, не налоговая консультация**; финальное решение —
  человек.

## 12. Тестируемость

- **Среда расчёта:** золотые тесты по сценариям (зарплата + RSU + дивиденды + продажа →
  ожидаемые строки), чтобы расчёт был доказуемо корректен.
- **Сгенерированные функции:** обязательные автотесты + sanity на граничных случаях перед
  одобрением.
- **Пайплайн документов:** фикстуры реальных форматов выписок (обезличенные) → ожидаемые сущности.
- **PII-граница:** тесты, что персональные данные не уходят в облачный путь (только токены).
- **Экспортеры:** валидация XML против XSD; снэпшот-тесты Summary/PDF.
- **Изоляция доменного слоя** за интерфейсами для подмены MAF/провайдера в тестах.

## 13. Ключевые риски и митигации

| Риск | Митигация |
|---|---|
| Неверная трактовка нормы агентом | Провенанс на каждую цифру; human-in-the-loop; спорное выносится на подтверждение |
| Ошибка в цифрах | Расчёт только через тестируемые функции, decimal, без float; золотые тесты |
| Утечка PII в облако | PII-middleware (псевдонимизация); локальный режим; раздельное хранилище |
| Исполнение недоверенного кода (генерированного/импортированного) | Песочница без сети + лимиты + тесты + одобрение + пиннинг |
| Prompt-injection из документов/скиллов | Трактовка как данные; извлечение по схеме; цифра обязана иметь проверяемый источник |
| Незрелость MAF API | Изоляция доменного слоя за собственными интерфейсами |
| Изменение закона/формы между годами | Версионирование корпуса/формы; пиннинг функций; инвалидация по версии |

## 14. Открытые вопросы (для уточнения при планировании)

- Конкретный дефолтный LLM-провайдер для первого прогона (Azure OpenAI vs Ollama) —
  параметризовано, но нужен дефолт для разработки.
- Источник официального XSD/структуры XML портала EPO (для вехи 3 экспорта).
- Источник и формат исходников законодательства для индексации (zakonyprolidi vs Sbírka zákonů)
  и политика обновления редакций.
- Технология песочницы на macOS (контейнер vs подпроцесс с ограничениями).
- Формат манифеста skill/knowledge pack (минимальная схема для v1).

## 15. Резюме решений (для трассировки)

1. Аудитория: личный инструмент + несколько похожих пользователей; не SaaS.
2. LLM-провайдер: параметризован (`IChatClient`); данные локально; PII-граница на облачном пути.
3. Выход: по нарастающей — Summary → PDF → XML.
4. Платформа/ядро: .NET + Microsoft Agent Framework + Microsoft.Extensions.AI + MCP + Spectre.Console.
5. Расчёт: сгенерированные, протестированные, одобренные функции (decimal); агент не считает float.
6. Источники правил: законодательство (RAG, без хардкода) + официальная форма/инструкция (DAG строк).
7. Примитивы decimal-математики: add/subtract/multiply/divide/round(CZ); граница «формальный расчёт — через функции».
8. Документы: автоклассификация, OCR/Vision для сканов, извлечение сущностей по схеме, генерация парсеров в песочнице.
9. Память: профиль/факты/фидбэк/артефакты/предпочтения; фидбэк приоритетен; инвалидация по версии.
10. Расширяемость: skills + MCP, без plugins.
11. Шеринг: раздельное хранилище + knowledge packs (= skills) через git-репо/PR; импорт через песочницу+тесты+одобрение.
12. Сквозное: PII-middleware, песочница, экспортеры поверх канонической модели, human-in-the-loop, дисклеймеры.
